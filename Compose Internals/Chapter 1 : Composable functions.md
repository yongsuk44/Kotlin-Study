## The nature of Composable functions

컴포저블 함수는 컴포즈의 원자 빌딩 블록(atomic building block)이자, 컴포저블 트리를 작성하는데 사용되는 구조 입니다.  
트리로 표현하는 이유는 컴포즈 런타임이 큰 트리의 노드로 컴포저블 함수를 메모리에 표현할 수 있기 때문입니다. 

기분 문법은 Kotlin 함수에 `@Composable` 어노테이션을 추가하는 것으로 컴포저블 함수를 만들 수 있습니다.

```kotlin
@Composable
fun NamePlate(name: String) { ... }
```

위와 같이 `NamePlate`의 데이터를 컴포저블의 트리의 노드로 변환할 것임을 컴파일러에게 알릴 수 있습니다.  

예를 들어, `@Composable (Input) -> Unit`는 다음을 의미합니다.
- `Input` : 데이터
- `Unit` : 요소를 메모리 내의 컴포저블 트리에 등록하는 작업

> `Input`을 받아들이는 함수에서 `Unit`을 반환하는 것은 함수 본문에서 `Input`을 소비하고 있다는 의미입니다.

이와 같은 작업은 컴포즈 용어로 '발행(emitting)'이라고 하며, 발행은 노드 트리에 변경 사항을 추가하는 것을 말합니다.  
발행 작업은 컴포저블 함수가 실행될 때 수행되며, 이는 컴포지션 단계에서 이루어집니다.

그러나, 모든 컴포저블 함수가 `Unit`을 반환하지 않습니다. 
일부 함수는 값을 반환하며, 이는 컴포저블 함수의 의미가 변경됩니다.
이와 같은 함수들은 `Input`을 "소비"하는 것이 아닌, `Input` 기반으로 값을 "제공"하는데 초점을 맞추고 있습니다.

대표적으로 `remember` 컴포저블 함수가 있습니다:

```kotlin
@Composable
fun NamePlate() {
    val name = remember { generateName() }
    Text(name)
}
```

`remember`는 어떤 연산의 결과를 메모이제이션(memoization)하고 이를 반환하는 역할을 합니다.  
또한, 트리의 인메모리 표현(in-memory representation)은 `remember`가 실행될 때마다, 이 함수의 호출 정보와 결과를 업데이트합니다. 
이는 함수 호출 그래프의 구조와 관련 정보를 항상 최신 상태로 유지하는데 중요한 역할을 합니다.

`@Composable` 어노테이션은 함수에 몇 가지 중요한 속성을 갖게하여 컴포즈 런타임이 다양한 최적화를 수행할 수 있게 합니다.
예를 들어, 병렬 컴포지션, 우선순위에 따른 임의의 컴포지션, 스마트 재구성, 위치 기반 메모이제이션 등의 최적화가 가능해집니다.

> 런타임 최적화는 실행할 코드에 대한 "몇 가지 확신"을 가질 때, 이를 바탕으로 특정 조건과 행동을 가정할 수 있게 됩니다.
> 이는 다양한 실행 전략이나 평가 기법을 활용하여 코드를 '소비(실행)'할 기회를 제공합니다.
> 
> "몇 가지 확신"의 예시 중 하나는 코드 내의 다양한 요소들 간의 관계를 들 수 있습니다.
> 만약, 코드 내 요소들이 의존적이지 않다면, 병렬로 실행할 수 있고, 순서를 바꿔도 결과에 영향을 주지 않는다면 임의의 순서로 실행할 수 있습니다.

## Calling context

컴포저블 함수는 컴포즈 컴파일러에 의해 암묵적으로 Composer 컨텍스트 인스턴스를 파라미터로 받는 함수로 변환되며, 
이 인스턴스를 자신의 컴포저블 자식에게도 전달합니다. 이는 컴포즈 런타임과 개발자가 명시적으로 다룰 필요 없이, 주입된 암묵적 파라미터로 생각할 수 있습니다.

예를 들어, 다음 컴포저블이 있다고 가정해봅시다:

```kotlin
@Composable
fun NamePlate(name: String, lastname: String) {
    Column(
        modifier = Modifier.padding(16.dp)
    ) {
        Text(text = name)
        Text(
          text = lastname,
          style = MaterialTheme.typography.body2
        )
    }
}
```

컴파일러는 트리의 각 컴포저블 호출에 암묵적인 Composer 파라미터를 추가하고, 각 컴포저블의 시작과 끝에 몇 가지 마커를 추가합니다.
다음 코드는 단순화된 예시이지만, 결과는 다음과 같을 것입니다:

```kotlin
fun NamePlate(name: String, lastname: String, $composer: Composer) {
    $composer.start(123)
    Column(
        modifier = Modifier.padding(16.dp)
    ) {
        Text(
          text = name,
          $composer
        )
        Text(
          text = lastname,
          style = MaterialTheme.typography.body2,
          $composer
        )
    }
    $composer.end()
}
```

이처럼 트리가 오직 컴포저블 함수로만 구성되어 있는 경우에 Composer 컨텍스트를 하위 컴포저블에서도 사용할 수 있도록 합니다.
이는 컴파일러가 매우 엄격한 규칙(컴포저블은 다른 컴포저블에서만 호출될 수 있음)을 적용하여 보장합니다.

이러한 엄격한 규칙을 통해, 컴포즈는 런타임에 필요한 정보가 어떤 서브트리에 있어도 항상 접근이 가능하도록 보장할 수 있습니다.
이를 기반으로, **컴포저블 함수는 컴포지션 동안 주입된 Composer 인스턴스를 통해 트리의 변경 사항을 발행하며, 
이후 발행된 변경 사항은 재구성(Recomposition)의 기준이 됩니다.**

이것이 개발자가 작성한 코드와 컴포즈 런타임을 연결하는 방법입니다.   
이 연결을 통해 런타임에는 트리의 형태를 알리고, 이를 바탕으로 인메모리 표현을 구축하여 다양한 최적화가 가능해집니다.

## Idempotent

컴포저블의 또 다른 속성은 프로그램 상태에 대해 멱등성(idempotent)을 유지해야 한다는 것입니다.  
이는 컴포저블에 동일한 입력 데이터로 1000번 호출해도 항상 동일한 프로그램 상태를 발행함을 의미합니다.

위 가정을 기반으로 컴포즈 런타임은 재구성과 같은 작업을 수행합니다.
 
> 재구성은 컴포저블이 의존하는 데이터가 변경될 때 해당 컴포저블을 재호출하여, 업데이트된 요소들을 발행하는 작업입니다.
> 이를 UI로 번역하면, 컴포저블은 UI 트리에 새로운 노드 또는 업데이트된 노드를 발행할 수 있습니다.

재구성은 여러 이유로 발생할 수 있으며, 하나의 함수가 여러 번 재구성될 수 있습니다.
이런 이유로, 컴포저블이 멱등성을 갖지 않는다면 재구성이 될 때마다 프로그램 상태가 부수 효과(side effect)으로 인해 변경될 수 있습니다.

재구성은 트리를 따라 내려가면서 어떤 노드가 재구성되어야 하는지를 확인하는 수직 작업입니다.

"스마트 재구성"은 트리의 일부가 변경된 데이터에 의존하지 않는 경우, 변경되지 않고 그대로 유지되어 재구성 되지 않는 것을 의미합니다.
이는 해당 컴포저블 람다들이 재호출되지 않음을 의미하기에 효율성 측면에서 큰 발전입니다.  
다시 생각해보면, **컴포저블 함수들이 방출한 상태가 이미 메모리에 저장되어 있고, 입력이 변경되지 않으면, 이를 재사용할 수 있기에 이런 최적화가 가능합니다.
이것이 멱등성의 직접적인 결과입니다.**

만약, 컴포저블이 동일한 입력에 대해 항상 동일한 프로그램 상태를 반환하지 않는다면, 런타임은 이러한 가정을 할 수 없으며 최적화를 수행할 수 없습니다.

## Free of side effects

"순수 함수" 또는 "순수성"이라는 용어는 부수 효과(side effects)가 없는 함수를 의미합니다.

> "부수 효과"를 함수의 범위를 벗어나, 예상치 못한 일을 하는 모든 작업으로 이해할 수 있습니다.  
> 컴포즈에서 부수 효과는 컴포저블의 범위를 벗어나는 프로그램의 상태 변경을 의미합니다.
> 더 넓은 의미로는 전역 변수를 설정하거나, 메모리 캐시를 업데이트하거나, 네트워크 쿼리를 수행하는 등이 부수 효과로 간주될 수 있습니다.
> 
> 실제로 함수가 네트워크 쿼리 실패, 함수가 여러 번 실행되는 동안 외부 캐시가 업데이트되는 상황 등 이런 행동에 의존하게 되면 
> 모호성(ambiguity)의 원인이 되고, 함수를 비결정적(non-deterministic)으로 만듭니다.
> 추가로 이런 부수 효과들은 간헐적으로 경쟁 조건(race condition)을 초래합니다.

컴포저블이 멱등성을 가지려면 통제되지 않은 부수 효과를 실행해서는 안됩니다.
그렇지 않으면, 매 실행(i.e : 컴포지션)마다 다른 프로그램 상태를 반환할 수 있기에 멱등성을 잃게됩니다.
이는 멱등성과 부수 효과가 매우 밀접하게 연결되어 있음을 의미합니다.

부수 효과를 허용하면 컴포저블이 이전 컴포저블의 실행 결과에 의존하는 문제가 되는 상황이 발생할 수 있습니다.  
이는 런타임이 컴포저블을 어떤 순서로든(심지어 병렬로) 실행 되기를 기대하기 때문에 절대로 허용해서는 안됩니다.  
재구성을 병렬로 수행하면 다른 스레드로 오프로드하고 여러 코어를 활용할 수 있습니다. 예를 들어 :

```kotlin
@Composable
fun MainScreen() {
    Header()
    ProfileDetail()
    EventList()
}
```

위 예시에서 `Header`, `ProfileDetail`, `EventList`는 순차적으로 실행되는 것이 아닌, 임의의 순서로 실행됩니다.

이런 관점에서 개발자들은 아래 상황과 같은 컴포지션의 순서에 의존하는 로직을 생성해서는 안됩니다.

- `ProfileDetail`에서 상태 업데이트 후, `EventList`에서 새로운 효과를 트리거
- `Header`에서 전역 변수를 설정하고, `EventList`에서 설정된 전역 변수 읽기

위와 같은 상황에서 앞뒤 순서가 바뀌면 프로그램이 어떻게 동작할지 예측할 수 없습니다.  
때문에, 컴포지션의 부수 효과를 통해 컴포저블 간의 관계를 설정하는 것은 잘못된 방식이며 피해야 합니다.  
비즈니스 로직이 필요하다면, 이는 하나 이상의 컴포저블의 책임으로 만드는게 아닌, 다른 아키텍처 계층으로 위임해야 합니다. 

> 컴포즈는 우선순위에 따라 컴포지션을 재배치할 수 있는 기능이 있습니다.  
> 예를 들어, 화면에 표시되지 않는 컴포저블에 낮은 우선순위를 할당할 수 있습니다.

컴포저블에서 직접 부수 효과를 실행하면, 재구성의 부수 효과로 인해 컴포저블이 여러 번 호출될 수 있습니다.  
이는 코드와 프로그램 상태의 무결성을 잠재적으로 손상시키고, 경쟁 조건을 만들기에 위험합니다.

예를 들어, 네트워크에서 데이터를 로드하는 컴포저블 함수를 생각해보면 다음과 같습니다:

```kotlin
@Composable
fun EventFeed(service: NetworkService) {
    val events = service.loadEvents()
    
    LazyColumn {
        items(events) { event ->
            EventCard(event)
        }
    }
}
```

여기서 효과(effect, 네트워크 호출)는 재구성될 때마다 다시 실행되며, 
매우 짧은 시간 내에 해당 컴포저블이 여러 번 재구성될 수 있어 제어나 조정 없이 많은 효과가 동시에 발생할 수 있습니다.

그렇지만, 상태를 갖는 프로그램을 만드려면 부수 효과가 필요하기에, 컴포즈는 컴포저블에서 부수 효과를 안전하게 호출할 수 있는 메커니즘을 제공합니다.
이 메커니즘은 컴포저블 생명주기를 인식하도록 하여, 재구성 중에 작업을 계속할 수 있게 해줍니다.
이를 효과 핸들러(effect handler)라고 하며, 이후 챕터에서 다룰 것입니다.

컴포저블에서 어떤 종류의 제어 없이 부수 효과를 실행할 때 발생 가능한 또 다른 문제의 예시로는, 외부 변수를 상태를 보유하는 컴포저블에서 변수를 업데이트하는 행동을 들 수 있습니다.
이는 컴포저블이 여러 스레드에서 실행될 수 있으므로, 당연하게도 해당 변수의 접근이 '스레드 안전(thread-safe)'하지 않기 때문에 문제가 발생할 수 있습니다.


```kotlin
@Composable
fun BuggyEventFeed(events: List<Event>) {
    var totalEvents = 0
    
    Column {
        Text(
            if (totalEvents == 0) "No events" 
            else "Total events: $totalEvents"
        )
        
        events.forEach { event ->
            Text(event.name)
            totalEvents++
        }
    }
}
```

여기서 `totalEvents`는 `Column`이 재구성될 때마다 수정됩니다.   
이는 총 개수가 일치하지 않고, 경쟁 조건에 노출됩니다.

> 마지막으로 흥미로운 점은 계속해서 "부수 효과가 없는(free of side effects)"것이 예측 가능한 코드를 작성하고 일관된 동작을 보장한다고 설명했습니다. 
> 그러나, 실제로 컴포지션을 생성하거나 업데이트하는 것은 컴포저블 호출 그래프를 실행하는 "부수 효과"입니다.
> 하지만 이 부수 효과는 라이브러리의 기능을 활성화하기에 구조적으로 허용되며, 프로그램의 동작에 어떤 방식으로든 일관성을 해치지 않기에 허용됩니다.

## Restartable

컴포저블은 재구성될 수 있으며, 이는 함수 체인의 일반적인 함수와 달리 여러 번 호출될 수 있음을 의미합니다.  
즉, 컴포저블 함수를 재구성한다는 것은 함수를 다시 실행한다는 것을 의미합니다.

컴포즈는 필요한 경우 언제든 컴포저블을 재구성하는 능력을 부여하여, 호출 그래프의 어떤 노드를 재구성할지 선택할 수 있습니다.
이는 본질적으로 관찰하는 상태의 변경에 따라 함수를 재실행시킬 수 있는 반응형 접근 방식입니다.

> 모든 컴포저블들은 기본적으로 재시작 가능하지만, 런타임에서는 `@NonRestartableComposable`을 제공하여 컴포저블이 재시작되지 않도록 할 수 있습니다.
> 즉, 이 어노테이션을 사용하면 컴파일러는 컴포저블이 재구성되거나 건너뛸 수 있는데 필요한 보일러 플레이트 코드를 생성하지 않습니다.
> 
> 이 어노테이션은 주로 다른 컴포저블에 의해 재구성(재시작)될 가능성이 높은 아주 작은 컴포저블에만 의미가 있을 수 있습니다.
> 이 작은 컴포저블들은 논리가 매우 적어 자체적으로 무효화하는 것이 큰 의미가 없기에, 이들의 무효화/재구성은 본질적으로 부모 또는 상위 컴포저블에 의해 관리될 가능성이 높습니다.

## Fast execution

컴포저블은 여러 번 호출될 수 있기에 빠르게 실행되어야 합니다.  
이러한 이유로 컴포저블은 실제 UI 생성이 아닌, 트리에 노드 변경 사항을 발행하는 방식으로 설계되었습니다.  
예를 들어, 컴포저블의 애니메이션은 매 프레임마다 호출될 수 있습니다. 
이와 같이 고비용 연산은 코루틴으로 오프로드되어야 하며, 항상 생명주기를 인식하는 효과 핸들러 중 하나로 래핑되어야 합니다. 
이처럼, 부수 효과를 생명주기에 맞추고 중단 상태로 만드는 것은 컴파일러가 이를 올바르게 사용하고, 런타임의 예상대로 조화를 이루도록 보장할 수 있는 방법입니다.

컴포저블과 컴포저블 트리는 프로그램의 설명을 작성하는 빠르고, 선언적이며, 경량화된 접근 방식으로 볼 수 있습니다.  
프로그램의 설명은 메모리에 유지되며 추후에 해석/구체화 됩니다.

## Positional memoization

위치 기반 메모이제이션(Positional memoization)을 이해하기 전, "함수 메모이제이션"에 대해서 먼저 알아야 합니다.  
함수 메모이제이션은 함수가 입력값에 따라 결과를 캐시하여, 동일한 입력에 대해 다시 계산할 필요가 없게 하는 기능입니다.
위에서 설명한 것처럼, 이는 순수(deterministic) 함수에 대해서만 가능합니다.   
순수 함수는 동일한 입력에 대해 항상 동일한 결과 반환이 확실하므로, 결과를 저장하고 재사용할 수 있습니다.

> 함수 메모이제이션은 함수형 프로그래밍 세계에서 널리 알려진 기법입니다. 
> 함수형 프로그래밍에서는 프로그램이 순수 함수의 조합으로 정의되므로, 함수의 결과를 메모이제이션하여 성능을 크게 향상시킬 수 있습니다.

위치 기반 메모이제이션(Positional Memoization)은 이 아이디어를 기반으로 하지만, 중요한 차이점이 있습니다.
런타임은 동일한 컴포저블에 대한 호출을 구분하기 위해 부모 내에서 컴포저블 호출의 위치를 기반으로 생성된 고유한 아이덴티티를 제공합니다.
즉, 컴포저블은 트리에서 자신의 위치에 대한 고유한 정보를 가지고 있습니다.  

런타임은 고유한 아이덴티티를 통해, 아래 예시와 같이 `Text()`에 대한 호출을 구분할 수 있습니다:

```kotlin
@Composable
fun MyComposable(txt: String) {
    Text(txt)
    Text(txt)
    Text(txt)
}
```

위 세 가지 `Text`는 동일한 입력을 가지고 있지만, 부모 내에서 서로 다른 위치에서 호출되기에, 컴포지션은 각기 다른 아이덴티티를 가진 세 개의 서로 다른 인스턴스로 처리합니다.
이 아이덴티티는 재구성 중에도 유지되므로, 런타임은 컴포지션을 통해 특정 컴포저블이 이전에 호출되었는지, 또는 변경되었는지를 확인할 수 있습니다.

때로는 아이덴티티의 생성이 호출 위치에 의존하기에 런타임은 아이덴티티 생성에 어려움을 겪을 수 있습니다.  
예를 들어, 동일한 컴포저블에 대한 여러 호출이 동일한 위치를 가지면서 다른 노드를 나타내는 경우입니다:

```kotlin
@Composable
fun TalksScreen(talks: List<Talk>) {
    Column {
        for (talk in talks) {
            Talk(talk)
        }
    }
}
```

여기서 `Talk(talk)`는 매번 동일한 위치에서 호출되지만, 각 `Talk`는 서로 다른 컴포저블로 취급되길 원합니다.  
이런 경우에 런타임은 '호출 순서에 의존'하여 아이덴티티를 생성하고 구분할 수 있습니다.

이 방식은 리스트의 끝에 새 요소를 추가할 때에는 나머지 컴포저블이 이전과 동일한 위치에 남아 있어 잘 동작하지만,
리스트의 상단이나 중간에 요소를 추가하게되면, 이전에 호출된 모든 컴포저블(`Talk`)들의 위치가 변경되기에 입력이 변경되지 않았더라도 재구성됩니다.
이러한 경우에는 비효율적이며 예기치 못한 문제를 초래할 수 있습니다.

이 경우에 런타임은 `key`를 사용하여 호출에 대한 아이덴티티를 수동으로 지정할 수 있습니다.

```kotlin
@Composable
fun TalksScreen(talks: List<Talk>) {
    Column {
        for (talk in talks) {
            key(talk.id) { // Unique Key
                Talk(talk)
            }
        }
    }
}
```

이렇게 하면 각 `Talk` 호출을 유니크할 가능성이 높은 `talk.id`와 연관 시킬 수 있으며, 
이를 통해 각 항목의 위치가 변경되더라도 컴포지션이 모든 항목의 아이덴티티를 유지할 수 있습니다.

"컴포저블이 컴포지션에 노드를 발행한다."는 것은, 컴포저블 호출에 관한 모든 정보가 컴포지션에 저장됨을 의미합니다.  
모든 정보에는 파라미터, 내부 컴포저블 호출, `remember` 호출의 결과 및 기타 관련 정보가 포함됩니다.  
이처럼 컴포지션에 정보를 저장하는 과정은 암묵적으로 주입된 Composer 인스턴스를 통해 이루어집니다.

컴포저블 함수는 자신의 위치를 알고 있으므로, 해당 위치에 대한 정보로 캐시된 값은 해당 위치의 컨텍스트 내에서만 저장됩니다.
보다 명확한 이해를 위해 다음 예시를 보겠습니다:

```kotlin
@Composable
fun FilteredImage(path: String) {
    val filters = remember { computeFilters(path) }
    ImageWithFiltersApplied(filters)
}

@Composable
fun ImageWithFiltersApplied(filters: Filters) {
    // Apply filters to the image
}
```

여기서는 이미지 경로를 기반으로 필터 계산(무거운 연산)의 결과를 `remember`로 캐시합니다.  
필터 계산 후, 이미 계산된 필터를 사용하여 이미지를 렌더링할 수 있습니다. 
이처럼 미리 계산된 결과를 캐싱하는 것은 성능 향상을 가져옵니다. 
**캐시된 값을 인덱싱하는 키는 "호출 위치 + 함수 입력(`path`)"을 기반으로 합니다.**

> `remember`는 슬롯 테이블에서 결과를 읽는 방법을 아는 컴포저블 입니다.  
> `remember`가 호출되면 테이블에서 해당 함수 호출을 찾아 캐시된 결과를 반환합니다.  
> 만약, 결과가 없다면 연산을 수행하고, 결과를 저장한 후 반환하여 나중에 재사용할 수 있도록 합니다.

`remember`는 위치 기반 메모이제이션을 활용하여, 이를 호출하는 컴포저블인 `FilterImage`에 의해 호출된 경우에만 캐시된 값을 가져옵니다.
이는 테이블로 가서 해당 컴포저블의 정보가 저장될 것으로 예상되는 슬롯 범위 내에서 값을 찾는다는 것을 의미합니다.

따라서, `remember`는 해당 범위 내에서만 단일 인스턴스처럼 작동합니다.  
즉, 초기 컴포지션 동안에만 값을 계산하고, 이후의 재구성에서는 캐시된 값을 가져옵니다. 
그러나, 동일한 컴포저블이 다른 컴포지션에서 사용되거나, 동일한 함수 호출이 다른 컴포저블에서 기억된(remembered) 경우, 기억된 값은 다른 인스턴스가 됩니다.

컴포즈는 위치 기반 메모이제이션 개념을 기반으로 구축되었으며, 이는 스마트 재구성의 기반이 됩니다.  
`remember`는 이를 명시적으로 사용하여 더 세밀한 제어를 가능하게 합니다.

## Similarities with suspend functions

Composable 함수가 호출 컨텍스트를 요구한다는 점은 Kotlin의 또 다른 언어 기본 기능인 suspend 함수와 유사합니다.

Kotlin에 익숙하면 suspend 함수 또한 호출 컨텍스트을 요구한다는 것을 알 것입니다.
즉, suspend 함수는 다른 suspend 함수에서만 호출될 수 있습니다. Kotlin 컴파일러는 이 규칙을 적용하여 suspend 함수 체인의 모든 함수를 새로운 버전으로 대체하고, 
이 버전은 각 단계에 암묵적인 파라미터를 전달합니다. 이 파라미터는 `Continuation`입니다.

```kotlin
suspend fun publishTweet(tweet: Tweet): Post = ...
```

위 코드를 Kotlin 컴파일러는 아래와 같이 변환합니다.

```kotlin
fun publishTweet(tweet: Tweet, continuation: Continuation<Post>): Unit
```

이때 Continuation은 컴파일러에 의해 추가되고 모든 suspend 호출에 전달됩니다.
이는 Kotlin 런타임이 다양한 suspend 지점에서 실행을 중단하고 재개하는데 필요한 모든 정보를 포함합니다.

suspend는 호출 컨텍스트를 강제하여 실행 트리를 통해 암묵적인 정보를 전달할 수 있는 방법을 보여주는 좋은 예입니다.

이런 의미에서, 우리는 @Composable을 언어 기능의 한 예로 이해할 수 있습니다.

> 여기서 생길 수 있는 합리적인 질문은 "Compose 팀은 왜 suspend를 사용하지 않았는가?" 입니다.
> 두 기능이 구현하는 패턴은 유사하지만, 언어에서는 완전히 다른 동작을 활성화합니다.
> 
> Continuation 인터페이스는 실행을 중단하고 재개하는 문제를 해결하는데 특화되어 있습니다.
> 이는 콜백 인터페이스로 모델링되며, Kotlin은 이를 위해 필요한 모든 메커니즘을 갖춘 기본 구현을 생성합니다.
> 반면, Compoes의 목표는 런타임에서 다양한 방식으로 최적화할 수 있는 큰 호출 그래프의 메모리 내 표현을 만드는 것입니다.
> 
> Compose 동작에서 필요한 점은 호출 컨텍스트를 강제하는 것뿐이며, 흐름 제어를 인코딩하는 것이 아닙니다.
> 흐름 제어 인코딩은 궁극적으로 Continuation이 하는 일입니다. 

## Composable functions are colored

Composable 함수는 일반 함수와 다른 기능과 속성을 가지고 있습니다. 이 함수들은 다른 타입을 가지며, 매우 특정한 관심사를 모델링합니다.
이러한 차별화는 일종의 "함수 컬러링"으로 이해할 수 있는데, 이는 별개의 함수 카테고리를 나타내기 떄문입니다.

함수 컬러링은 구글의 Dart 팀에서 작성한 블로그 포스트(What color is your function?)에서 설명한 개념입니다.
여기서는 async 함수와 sync 함수가 함께 잘 조합되지 않는다는 것을 설명하면서, 
sync 함수에서 async 함수를 호출하려면 sync 함수를 async로 만들거나 async 함수를 투명하게 호출하고 결과를 기다릴 수 있는 메커니즘을 제공해야 한다고 설명합니다.
그래서 일부 라이브러리와 언어에서는 조합성을 되찾으려 시도하기 위해 Promise와 async/await가 도입되었습니다.

Kotlin에서는 suspend가 이 문제를 어느 정도 해결하지만, 채색(colored)되어 있습니다.
suspend 함수는 특정한 실행 문맥(Continuation)이 필요하기에, 오직 suspend 함수에서만 suspend 함수를 호출할 수 있습니다.

이 문제의 해결 여부와 상관 없이, 이것이 일어나는 이유는 두 가지 다른 카테고리의 함수을 모델링하고 있기 때문입니다.
이는 두 개의 다른 언어를 사용하는 것과 같으며, 대부분의 경우 이들을 결합하는 것은 좋지 않은 결과를 초래합니다.
즉, 즉시 결과를 계산하는 연산(sync)과 시간이 걸리는 연산(async)이 있습니다.
후자는 파일 읽기, 데이터베이스 쓰기 또는 네트워크 서비스 쿼리와 같은 시간이 걸리는 작업에 주로 사용됩니다.

Composable 함수들은 불변 데이터를 트리의 노드에 매핑하는 재시작 가능하고 메모이제이션 가능한 함수를 나타냅니다.
Compoes 런타임은 이 사실에 의존합니다. 이것이 컴파일러가 Composable 함수가 다른 Composable 함수에서만 호출되도록 강제하는 이유입니다.
이러한 제한은 컴포지션을 활성화하고 구동하는 호출 컨텍스트를 보장하기 위해 사용됩니다.

하지만, "함수 컬러링"의 근본적인 문제는 채색(colored)이 서로 다른 함수를 투명하게 조합할 수 없다는 점에서 비롯됩니다.
Compose에서는 Composable 함수는 Non-composable 함수에서 호출될 것으로 기대되지 않는데, 이는 Compose를 사용하는 방식이 Composable 함수들로만 그래프를 구축하기 때문입니다.
이들은 개발자가 트리를 구축하는데 사용하는 DSL의 기본 단위입니다. Composable 함수는 프로그램 로직을 작성하거나 일반 함수와 조합하기 위해 고안된 것이 아닙니다.

여기서 약간의 트릭이 존재함을 알 수 있습니다. Composable 함수의 장점 중 하나는 로직을 통해 UI를 선언할 수 있다는 점입니다.
이는 때때로 일반 함수에서 Composable 함수를 호출해야 한다는 것을 의미합니다.

예를 들어:

```kotlin
@Composable
fun SpeakerList(speakers: List<Speaker>) {
    Column {
        speakers.forEach {
            Speaker(it)
        }
    }
}

@Composable
fun Speaker(speaker: Speaker) {
    Text(speaker.name)
}
```

여기서 `forEach` 람다에서 `Speaker`를 호출하고 있으며, 컴파일러는 이에 대해 불평하지 않습니다.

이처럼 함수 색상을 섞어 사용할 수 이유는 `inline`에 있습니다.
컬렉션 연산은 모두 `inline`으로 지정되어 있어, 이들은 본질적으로 자신의 람다를 호출자에게 inline합니다.
위 예제에서 `Speaker` 호출은 `SpeakerList` 본문 내에서 inline되며, 이는 둘 다 Composable 함수이기 때문에 허용됩니다.

따라서 API에서 inline을 활용함으로써 Composable 함수의 로직을 작성할 때 함수 색 구분 문제를 피할 수 있습니다.
Composable 함수 자체는 여전히 호출 컨텍스트를 강제함으로써 색이 구분됩니다.
또한, Composable에서 기대되는 로직 타입은 보통 조건부 로직으로, 이는 특정 조건(보통 어떤 상태나 파라미터)에 따라 Composable을 교체하는 로직입니다.
이러한 로직은 대부분 단순하며, inline으로 작성되거나 필요한 경우 inline 함수로 분리할 수 있습니다.

Composable 함수가 색을 가지고 있다는 것을 이해하면, 이들이 반드시 다른 카테고리의 함수가 되어야함을 이해할 수 있습니다.
이를 통해 Compose 컴파일러가 이들을 다르게 처리하고, 이들의 동작과 호출 컨텍스트에 대한 가정을 통해 이들의 능력을 활용할 수 있습니다.

## Composable function types

`@Composable` 어노테이션은 함수의 타입을 컴파일 시에 변경합니다. 더 구체적으로, Composable 함수는 `@Composable (T) -> Unit` 함수 타입을 따릅니다. 
여기서 `T`는 입력 데이터이고 `Unit`은 함수가 입력을 받아 트리에 변화를 주어야 함을 나타냅니다. 
개발자는 이 타입을 사용하여 표준 람다처럼 Composable 람다를 선언할 수 있습니다.

```kotlin
// This can be reused from any Composable tree
val textComposable: @Composable (String) -> Unit = { 
    Text(
        text = it,
        style = MaterialTheme.typography.body1
    )
}

@Composable
fun NamePlate(name: String, lastName: String) {
    Column(
        modifier = Modifier.padding(16.dp)
    ) {
        Text(
            text = name,
            style = MaterialTheme.typography.h6
        )
        textComposable(lastName)
    }
}
```

Composable 함수는 또한 수신자(receiver)를 가지는 람다가 필요할 때 `@Composable Scope.() -> Unit` 타입을 따를 수 있습니다.
이는 Composable DSL을 활용하여 Composable을 중첩할 수 있도록 하는 스타일로 자주 사용됩니다.
이러한 경우, 스코프는 일반적으로 modifier나 해당 Composable 내에서 읽을 수 있는 관련 변수를 포함합니다.

Composable 함수는 수신자(receiver)를 갖는 람다가 필요할 때 `@Composable Scope.() -> Unit` 타입을 따를 수도 있습니다. 
이는 Composable DSL을 사용하여 Composable 함수를 중첩할 수 있게 합니다. 
이러한 경우, 스코프는 보통 modifier나 해당 Composable 내에서 사용할 변수를 포함합니다.

```kotlin
inline fun Box(
    ...,
    content: @Composable BoxScope.() -> Unit
) {
    // ...
    Layout(
        content = { BoxScopeInstance.content() },
        measurePolicy = measurePolicy,
        modifier = modifier
    )
}
```

또한, Compose 컴파일러는 Composable 함수에 특정한 제한 사항과 속성을 부과합니다. 
이러한 요구 사항은 Composable 함수의 타입 정의의 일부로 볼 수 있습니다. 
타입은 데이터를 세분화하고 속성 및 제한 사항을 부여하여 컴파일러가 이를 정적으로 검사할 수 있게 합니다. 
이는 @Composable 어노테이션이 하는 일입니다.

Composable 함수의 요구 사항은 코드를 작성하는 동안에도 Compose 컴파일러에 의해 검사됩니다. 
컴파일러는 호출, 타입 및 선언 검사기를 사용하여 필요한 호출 문맥을 보장하고, 멱등성을 유지하며, 통제되지 않은 부작용을 방지합니다. 
이러한 검사기는 Kotlin 컴파일러의 프론트엔드 단계에서 실행되며, 이 단계는 정적 분석에 사용되며 가장 빠른 피드백을 제공합니다. 
이 라이브러리는 의도적으로 설계된 공개 API와 정적 검사를 통해 개발자에게 안내된 경험을 제공합니다.
